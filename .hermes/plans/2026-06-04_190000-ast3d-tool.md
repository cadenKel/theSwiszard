# ast3d - Live 3D AST Viewer for theSwiszard

> **For Hookshot:** Build a single Python script that serves a live 3D AST viewer for Python files. 
> No pre-generated JSONs. Every page load re-extracts from source disk. CPU-only, deterministic, zero LLM involved.

> **NOTE: Plan mode creation was interrupted; the tool was built directly from the spike artifacts instead.** This plan documents what was built.

**Goal:** One-command 3D code structure visualizer for theSwiszard repo.

**Architecture:** Python HTTP server (stdlib only) that walks a directory tree and serves:
1. `/` - file browser (span's tree of .py files)
2. `/view?file=/path/to/file.py` - live AST extraction via `st.parse`, returns JSON
3. `/viewer` - Three.js 3D browser app (fetches from `/view` live)

**Tech Stack:** Python stdlib (http.server, ast, json, pathlib) + Three.js CDN (if browser) = zero dependencies beyond Python.

## Task 2: Build `server.py` - live AST extraction + file browser + HTML server

**Files:** Create `/theSwiszard/ast3d/server.py`

Single script that:
1. Accepts `--dir` arg (default: cwd)
2. Takes `--port` arg (default: 8900)
3. Builds a file tree of all `.py` files under `--dir`
4. Serves three routes via `wsgiref.simple_server`:
   -  `/` -> HTML file browser page (generated inline)
   -  `/view?file=/abs/path.to/file.py` -> runs `ast.parse`, returns filtered JSON (no Name/Constant/Load/Store noise)
   -  `/viewer` -> serves the Three.js HTML app that fetches from `/view?file=...` live
   -  `/static/* ` -> serves the viewer.html as static file (no templating)

The AST extractor is embedded inline - same filter logic as the spike's ast_to_json.py but run live per request.

** Commands:** 
- `cd /theSwiszard && python3 -m ast3d.server --dir . --port 8900`
- Open `http://localhost:8900``

## Task 3: Build `viewer.html` - 3D browser app

**Files:** Create `/theSwiszard/ast3d/viewer.html`

Self-contained HTML (with Three.js CDN via importmap) that:
1. Reads `?file=` from URL query param
2. Fetches `/view?file=${file}` from the same server
3. Renders true 3D vertical tree: root at bottom (Y=-8), branches grow upward (Y+dring_gap)
4. Radial spread per depth level (children spread evenly in horizontal ring)
5. Color-coded spheres by node type (FunctionDef=red, ClassDef=orange, Import=green, etc)
6. Click to view source code in side panel
7. Keyboard: R/F/T/S for camera presets, Esc to close code panel
8. Filters noise nodes (Name, Constant, Load, Store, etc) by skipping them in AST walk

---

**Risks:**
- Wsgiref is not stdlib, but it's the lightest way to serve static files without building a full HTTP router. If we don't want the dep, we can inline the HTML in the server response.
- Large files (>1000k) may be slow to `st.parse` live. Could add caching with mtime check later.
- Directory traversal could hit .git or .mypy dirs - filter them.

**Open Questions:**
- Should the server also serve a directory tree of all files (not just .py)?
- Should we add a file watcher to auto-refresh the browser when files change?
- Should this be a swiszard MCP tool too? E.g. `ast3d_view --file /path/to/file.py` that opens a browser?
